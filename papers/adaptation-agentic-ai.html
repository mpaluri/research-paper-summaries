Adaptation of Agentic AI: A Survey of Post-Training, Memory, and Skills

Jiang, Lin, Shi, Wang, He, Wu, Zhong, Song, Zhang, Wang et al. (34 authors) — December 2025 (v3: March 2026)

📄 arXiv:2512.16301  ·  📥 PDF  ·  💻 GitHub

TL;DR: A unified 2x2 framework for making AI agents better after pre-training. Four paradigms — A1, A2 (adapt the agent), T1, T2 (adapt the tools) — organize 100+ methods across post-training, memory, and skills. Key finding: training smarter tools (T2) can match full agent retraining with 70x less data.

Level 1 — Beginner

▼

What is this paper about?

AI assistants like ChatGPT or Claude can chat and answer questions. Now imagine giving them the ability to do things — search the web, run code, look up a calendar, query a database. That's an "AI agent." It doesn't just talk; it acts.

The problem: even the smartest AI models are clumsy agents out of the box. They call the wrong tool, forget what they were doing, or fall apart on multi-step tasks. This paper asks: how do you make these agents better after they're built?

The student-with-a-toolbox analogy

Core idea

Think of an AI agent as a smart student sitting an open-book exam. The student (the AI model) has a toolbox on the desk — a calculator, a dictionary, a search engine, a notebook for memory. There are two fundamentally different ways to help:

1. Coach the student — teach them to reason better, use tools more skillfully, plan ahead.

2. Upgrade the toolbox — give them a better calculator, a smarter search engine, a better notebook.

The four strategies (A1, A2, T1, T2)

The paper's big insight is that the entire messy landscape of "making agents better" boils down to a clean 2x2 grid based on two questions: What are you improving? (student vs. toolbox) and What signal tells you it's working? (did the tool work, or did the final answer come out right?)

A1 — Coach the student using tool feedback

The student writes code, runs it, sees "all tests pass." That success signal teaches the student to write better code next time. Like learning from lab experiments — the equipment gives you direct feedback.

A2 — Coach the student using final-answer feedback

Did the final answer on the exam match the answer key? If yes, reinforce whatever the student did. Like grading a math test — you only care about the bottom line.

T1 — Upgrade the tools independently

Build a better search engine or calculator that any student can use. The tools are trained on their own, without any specific student in mind. Plug-and-play.

T2 — Upgrade tools for this specific student

The student is fixed (maybe a closed-source API you can't retrain). But you can tune the search engine to return results this particular student understands best. The student's performance tells you whether the tool upgrade worked.

Why does this matter?

The paper argues that the best strategy is often not to retrain the whole AI model (expensive, risky, causes forgetting). Instead, upgrade the tools around it. A striking finding: a small 7B-parameter search tool trained with just 2,400 examples using T2 matched or beat a monolithic agent trained with 70x more data.

Key takeaway

Real-world systems will combine strategies: rare, expensive retraining of the core model (A1/A2) with frequent, cheap upgrades to tools, memory, and helper modules (T1/T2). You don't retrain your CEO every week — you upgrade the tools and processes around them.

Quiz — Level 1
1. What is the core problem this paper is trying to solve?
The paper focuses on adaptation — improving agents, tools, and their interaction after pre-training, not building models from scratch.
2. In the student-with-a-toolbox analogy, what does the "toolbox" represent?
Tools are external callable components — retrievers, APIs, code executors, memory modules — that the agent (student) uses to extend its capabilities beyond its internal knowledge.
3. What makes A1 different from A2?
Both A1 and A2 adapt the agent. The difference is the signal source: A1 learns from tool execution outcomes (e.g., code tests passed), while A2 learns from whether the final answer was correct.
4. Why is T2 particularly important for closed-source AI models?
T2 keeps the agent frozen and optimizes the tools around it using the agent's outputs as a signal — perfect when you can't touch the model's parameters.
5. What was the striking data efficiency finding about T2 vs. A2?
The s3 system (a T2 approach) trained a 7B search subagent with just 2,400 examples and matched agents trained with 70x more data — showing T2's dramatic data efficiency.

Level 2 — Intermediate

▼

The mathematical setup

Every adaptation method optimizes some objective function. The paper defines three ingredients: Agent A(theta) — the foundation model parameterized by theta; Tool T — everything external the agent calls; Objective O — a measurable signal. A1 and A2 both update theta but differ in what O measures. T1 and T2 leave theta frozen and update tool parameters phi, differing in where the training signal comes from.

A1 in practice: learning from tool execution

The RLVR progression

Toolformer (2023) — Self-supervised SFT. Inserts API calls into text, keeps calls that reduce model perplexity.

Gorilla (2024) — AST-based correctness checking of generated API calls against ground-truth schemas.

DeepRetrieval (2025) — Full RL. Query reformulation as an MDP with reward = alpha*Recall@k + beta*nDCG@k + gamma*format_score. Trained with KL-regularized PPO. Achieved 3x recall improvement (65.1% vs 24.7%).

A2 in practice: learning from final outputs

A2 doesn't care how tools were used — only whether the final answer was right. Subtle trap: the agent can learn to ignore tools entirely and still improve if the answer happens to be correct.

Three branches

Reasoning-centric RL (DeepSeek-R1) — Pure RL with verifiable output rewards, no SFT warm-start needed. Emergent chain-of-thought and self-correction.

Self-refinement (TextGrad) — LLM critiques its own output, treats critique as a "textual gradient." Parameter-free — works on black-box APIs.

Multi-tool orchestration (ReTool) — Trains agents to decide when and how to use tools via outcome-based reward with reflection and multi-stage rollouts.

T1: plug-and-play tools

T1 tools are trained independently and serve any agent: CLIP and SAM for vision, dense retrievers for search. A clever insight: an agent trained via A1 (like DeepRetrieval) can be frozen and reused as a T1 tool — the paper calls this "graduation."

T2: the "symbiotic inversion"

Key T2 methods

REPLUG — Trains a retriever by using the frozen LLM's perplexity as the reward. Documents that reduce the LLM's confusion get ranked higher.

s3 — 7B search subagent trained with "Gain Beyond RAG" reward: G = Quality(answer_with_search) - Quality(answer_with_RAG_only). Only 2,400 examples needed. Matched agents requiring 170K+ examples.

Memory as T2 — Long-term memory is a special case of T2: an external store with learned read/write functions, updated using signals from the frozen agent. Memento trains memory policies from binary outcome rewards alone.

Key takeaway

The paper recommends rare, expensive A1/A2 updates to establish a strong base model, then frequent, cheap T1/T2 adaptation of retrievers, search policies, memory, and planners for robustness and scalability.

Quiz — Level 2
1. Why can A2 methods accidentally train the agent to ignore its tools?
Since A2 supervises only the final output, the agent can learn strategies that improve answer quality without invoking tools — the training signal doesn't explicitly reward tool use.
2. How does DeepRetrieval frame query reformulation, and what makes it A1?
DeepRetrieval treats query rewriting as an MDP where the reward combines retrieval metrics (Recall, nDCG) and format correctness — all derived from tool execution, the defining feature of A1.
3. What makes s3 (T2) so data-efficient compared to monolithic A2 training?
T2 localizes adaptation to a small module (7B search subagent) learning one specific skill. A2 must update billions of parameters across entangled capabilities, requiring far more data.
4. How can an A1-trained agent become a T1 tool?
The paper calls this "graduation" — an agent trained via A1 (e.g., DeepRetrieval) can be frozen and reused as a plug-and-play T1 tool in entirely new agentic systems.
5. What is "parasitic adaptation"?
Parasitic adaptation is a security risk where a compromised tool or environment manipulates the RL reward signal to hijack the agent's learning process — effectively controlling what the agent learns.

Level 3 — Expert

▼

Formal framework

Adaptation is defined over a tuple (A, T, D, E, O): Agent A(theta), Tool set T, offline Data D, Environment E, and Objective O. The four paradigms are distinguished by two binary choices:

A1: min_theta O_tool(T(A_theta(x)))      -- agent weights, tool-execution signal
A2: min_theta O_agent(A_theta(x, T(a)))   -- agent weights, output signal
T1: min_phi   O_tool(T_phi(a))            -- tool weights, independent signal
T2: min_phi   O_agent(A(x, T_phi(a)))     -- tool weights, frozen-agent signal

When a method modifies both agent and tool, the paper assigns it to the paradigm of its dominant locus of optimization and flags the secondary component explicitly.

A1: the RLVR evolution

Three phases

Phase 1 — Toolformer (2023): Self-supervised SFT. Keep tool call c if Loss(text_with_c) < Loss(text_without_c). Perplexity reduction is the signal.

Phase 2 — Gorilla (2024): AST-based correctness. Parse generated API call into an abstract syntax tree, compare against ground-truth schema.

Phase 3 — DeepRetrieval (2025): Full MDP. R = alpha*Recall@k + beta*nDCG@k + gamma*format_score. Policy: max E[R(s,a)] - beta*KL(pi_theta || pi_ref). Result: 65.1% vs 24.7% recall.

Cross-domain principles: (1) verifiable rewards essential, (2) reward must be dense enough for useful gradients, (3) KL regularization prevents mode collapse.

A2: DeepSeek-R1 and TextGrad

DeepSeek-R1: Pure RL with verifiable rewards (no SFT warm-start). Emergent chain-of-thought, self-correction, backtracking. Sparse signal (final answer correctness) sufficient at scale.

TextGrad: Parameter-free A2. o_(t+1) = LLM(x, o_t, critique(o_t)). Gradient descent in natural language space. Works on black-box APIs — test-time compute scaling.

ReTool: Bridges A1/A2. Trains tool-use decisions via outcome reward (A2), but tool execution feeds into reasoning (A1-like). Classified A2 by dominant locus.

T2: REPLUG, s3, and memory

Formal T2 objectives

REPLUG: min_phi E_x[-log P_theta(o | x, R_phi(x))], theta frozen. Retriever optimized via frozen LLM perplexity. No parameter access needed — only output probabilities.

s3: Gain Beyond RAG reward: G = Quality(answer_with_search) - Quality(answer_with_RAG_only). Measures marginal value of search tool. 2,400 examples matched 170K+ example baselines.

Memento: Memory read/write policies trained from binary agent outcome rewards. Pure T2, no human annotation.

Memory taxonomy (v3 contribution)

  • Dynamic stores — episodic buffers, key-value caches
  • Experiential/reflective — Reflexion-style failed-attempt storage
  • Structured — knowledge graphs, trees, databases with learned read/write
  • Parametric — LoRA adapters / soft prompts (blurs agent/tool boundary)
  • Skill libraries — reusable procedural knowledge; skills from one agent become T1 tools for others

Connections to related work

Positioning

Vs. agent surveys (Wang 2024, Xi 2025): Those catalog architectures; this focuses on adaptation. The 2x2 taxonomy is novel.

Vs. RLHF: A2 is more general — includes non-parametric methods (TextGrad) and tool-augmented settings.

Vs. RAG surveys: Decomposes RAG into T1 retriever / T2 retriever / A1 generator / A2 generator — clarifying design decisions.

Vs. Claudini: Autoresearch is A1 (benchmark results = tool execution signal), with resulting optimizers deployable as T1 tools.

Critical evaluation

Strengths and weaknesses

Strengths: Genuinely clarifying taxonomy; concrete T2 efficiency case study (s3); honest boundary-case treatment; v3 adds memory, skills, and evaluation framework.

Weaknesses: 70x claim drawn from single comparison with potential confounders; no controlled experiments (survey, not experimental paper); paradigm boundaries can be ambiguous; scoped to single-agent only; evaluation framework prescriptive but not empirically validated; no formal inclusion/exclusion criteria for surveyed papers.

Key takeaway

The 2x2 framework (agent vs. tool, execution vs. output signal) provides a genuine organizing principle for agentic adaptation. The strongest practical insight is that T2 — training tools under frozen agent supervision — achieves remarkable data efficiency and modularity, making it the recommended default for production systems built on closed-source foundation models.

Quiz — Level 3
1. How does the paper formally distinguish A1 from A2?
Both A1 and A2 optimize theta (agent weights), but A1's objective is defined over the tool's execution result — O_tool — while A2's objective is defined over the agent's final output — O_agent. This is the formal axis that separates them, regardless of SFT vs RL.
2. What is the "Gain Beyond RAG" reward used in s3, and why is it effective?
By subtracting RAG-only quality from search-augmented quality, the reward isolates exactly how much the search subagent contributed beyond basic retrieval. This marginal signal prevents the searcher from getting credit for what the retriever already handles.
3. Why does the paper identify a "T2 evaluation gap"?
Benchmarks like BEIR measure retrieval in isolation (T1), and MMLU measures agent answers (A2), but no standard benchmark asks: "did this tool get better at serving this specific agent?" T2's defining property lacks dedicated evaluation infrastructure.
4. How does TextGrad achieve A2 adaptation without accessing model weights?
TextGrad frames LLM self-critique as gradient descent in natural language space. Since it only needs text outputs (not parameters), it works on any API — making it test-time compute scaling rather than traditional training.
5. What criterion does the paper use when a method modifies both agent and tool?
The paper uses "dominant locus of optimization" — whichever component receives the primary parameter updates determines the paradigm. The secondary component is noted. This is pragmatic and the authors acknowledge it can be ambiguous in edge cases.
